Executive Summary
Compliance-first property discovery platform for Dubai shared housing
What is NestMatch UAE?
▶

NestMatch UAE is a compliance-first property discovery platform solving Dubai's grey market shared housing crisis. We list ROOMS not bed-spaces — maximum 1 tenant per bedroom, municipality-permitted properties only, bilingual DLD agreements, and full KYC/AML verification.

The Problem: Grey market bed-space listings on Dubizzle, PropertyFinder, Bayut, and Facebook Marketplace violate Dubai Law No. 4 of 2026 (housing occupancy standards). Thousands of tenants live in unregulated shared bedrooms with no legal protections, no lease agreements, and no recourse.

Our Solution: NestMatch verifies every property, enforces 1 tenant per room, generates DLD-compliant bilingual viewing agreements, and builds a trust layer through tiered verification (UAE PASS OAuth), occupant profiles, and GCC Score matching. Landlords, agents, and tenants all benefit from compliance and transparency.

🔴 Hard Constraints (Never Violate)
▶

🚫 No payment processing — Requires CBUAE licence (out of scope)

🚫 No lease/tenancy contract drafting — Requires RERA broker licence (out of scope)

🚫 No escrow, deposit holding, rent collection — Financial services licence required

🚫 Star-only ratings (no text) — UAE Cybercrime Law § 34 (text reviews = defamation risk)

🚫 No twin share, bed-space, or shared bedroom listings — Law No. 4 of 2026 violation. maxLegalOccupancy = number of bedrooms (1 person each).

🚫 Identity fields always masked — Emirates ID, passport, UAE PASS ID hidden in UI. KYC docs only in R2 KYC_DOCS bucket, never IMAGES bucket.

🚫 Never set verification_tier to 'gold' without UAE PASS OAuth completing

▶

Frontend: React 19 + TypeScript + Vite → Auto-deploy on Vercel

Technical Architecture
Modern, serverless, state-machine driven, zero single point of failure
🎨 Frontend Stack
▶

Framework: React 19 (latest with new hooks)

Language: TypeScript (strict mode, zero-tolerance on type errors)

Build Tool: Vite (fast HMR, optimized production builds)

Deployment: Vercel (auto-deploy on git push)

Key Components: Navbar, ChatPanel, ViewingsPanel, ViewingAgreementModal, LeaseHandoffCard, DemoControls, Toast, UAEPassOverlay, PassportKycModal, ComplianceFlow, InboxBadge, ProtectedRoute

State Management: React Context (AuthContext, DemoStateContext, ToastContext)

API Strategy: Smart fallback to mock data if backend unreachable (3s health check, 30s cache)

⚙️ Backend Stack
▶

Runtime: Cloudflare Workers (serverless, edge-first, global CDN)

Web Framework: Hono (lightweight, TypeScript-first, fast routing)

Database: Cloudflare D1 (SQLite, serverless, automatic backups)

Object Storage: Cloudflare R2 (KYC_DOCS bucket, IMAGES bucket, cost-optimized)

Authentication: Custom JWT + OAuth 2.0 (Google, UAE PASS)

Database Tables: 24 tables including state machine audit logs (occupancy_events, viewing_events, verification_events, tenancy_events)

API Routes: 28+ routes covering auth, properties, users, viewings, agreements, KYC, occupancy, GCC, ratings, payments, inbox, Ejari, maintenance, chat

🔄 State Machines
▶

Every major entity is a state machine. Every transition is audit-logged.

Room Occupancy States:

available
→
pending_approval
→
occupied
→
notice_given
→
available

Every transition writes to occupancy_events audit log. Never go below 0 or above maxLegalOccupancy.

Viewing States:

PENDING → CONFIRMED → AGREEMENT_SENT → AGENT_SIGNED → FULLY_SIGNED → COMPLETED | NO_SHOW | CANCELLED

Every state transition writes to viewing_events. FULLY_SIGNED triggers room_occupancy → pending_approval.

User Verification States:

tier1_unverified → tier0_passport → tier2_uae_pass

Every state transition writes to verification_events. KYC documents stored only in R2 KYC_DOCS bucket.
